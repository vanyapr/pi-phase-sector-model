# pi-fs experiment 011: anchor-relative negative indexing

## Goal

Test the idea:

```text
If a finite prefix of pi repeats later, that repeat can be used as an anchor.
Once an anchor is known, negative offsets are well-defined.
```

This is not first-occurrence prediction. It is a file-system style address mode:

```text
(anchor pattern, occurrence number, relative offset, length) -> data
```

The key point is that reading before an anchor is just ordinary slicing relative
to a certified anchor address. In phase language, it corresponds to selecting an
inverse branch / left context.

## Interpretation

- A repeated finite prefix gives a valid anchor.
- Negative offsets are mathematically harmless inside a finite indexed horizon.
- The left context is the inverse branch information.
- Different occurrences of the same anchor usually have different left contexts.
- Therefore, for storage/addressing, first occurrence is not necessary; an anchor
  plus an occurrence number plus a relative offset is a valid address.

## Boundary

This experiment does not prove that every prefix repeats infinitely often.
It only demonstrates the anchor-relative addressing mechanism inside a computed
finite horizon.

## Proposed API

```python
anchor = locate_anchor(prefix, occurrence=k)
data = read_relative(anchor, offset=-128, length=256)
```

A proof object should include:

```text
anchor sector certificate
occurrence number
relative offset
direct phase/index slice certificate
```

This extends the finite locate contour from absolute pattern search to relative
navigation around certified anchor points.
